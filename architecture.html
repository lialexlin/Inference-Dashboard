Architecture

How the dashboard is wired

Static frontend, Python ETL backend, JSON in between. The browser fetches data/*.json directly — no build step, no API server, no framework. Everything below is a description of what runs and what each file is for.

What each step actually does — in plain English

No computer-science background needed. Read left-to-right, the way the data flows.

Step 2 · Adapters

"Talk to each website / database, bring the data home"

Every data source speaks a different language. Yahoo Finance gives you stock prices through a Python library. S&P Capital IQ wants you to log into a database and run a SQL query. SEC EDGAR has a public web URL that returns a list of filings. RSS feeds are XML files that need to be parsed.

An "adapter" is just a small Python file whose only job is to talk to one source. It logs in (if needed), makes the request, deals with errors, strips out junk (HTML tags inside RSS articles, for example), and hands back a clean Python dictionary.

Why split them up? So when Yahoo changes their API or Capital IQ adds a new field, you fix one small file. The rest of the system doesn't know or care.

Files: jobs/sources/prices.py · capital_iq.py · edgar.py · rss.py · openrouter.py · transcripts.py
Step 3 · Tag & score

"Turn raw data into meaning"

Raw data is just numbers and headlines. This step adds opinion:

  • Tagging — when a news article mentions "HBM3" or "CoWoS", which of the 15 layers of the stack does that belong to? jobs/transform.py reads a list of keywords per layer and stamps each article with its layer(s). It also flags "architectural risk" articles (anything that could make today's GPUs obsolete — Mamba, mixture-of-experts, etc.).
  • Bottleneck score — for each layer, how often are companies in it saying "supply constrained / sold out / lead times extending"? jobs/bottleneck.py counts those phrases over the last 90 days and produces a 0–100 number. High = the bottleneck is binding right now.
  • Business × Valuation score — for each ticker, is it growing fast (Business score) and is it cheap relative to peers (Valuation score)? Two numbers, plotted on a 2×2 grid that gives you the green/red labels you see on every chip. jobs/scoring.py does this. Full walkthrough in the next section ↓
  • Exit triggers — four pre-defined "if X happens, the whole thesis breaks" conditions. jobs/exit_triggers.py checks them and lights up green / amber / red.

None of this requires AI. It's keyword matching, arithmetic, and statistics. The judgement lives in what we measure (the keyword lists, the thresholds) — not in any black box.

Step 4 · Persist

"Save it as JSON files so the browser can read them"

"Persist" in software just means write to disk so it survives after the program stops running. A typical web app would save into a database (Postgres, MySQL). This dashboard is deliberately simpler — it writes plain text files in the data/ folder, one per topic. JSON is the file format; it looks like nested lists and dictionaries.

That's it. No database server. No login. When you open the website, your browser downloads those JSON files directly the same way it downloads images. The site can be hosted on free GitHub Pages because the "backend" is just static text.

Two safety properties worth knowing:

  • Atomic writes — instead of writing directly to scores.json, the code writes to scores.json.tmp, then renames it in one operation. If the script is interrupted mid-write, you still have yesterday's file intact. The browser never sees a half-finished file.
  • Partial preservation — if today the Capital IQ database is down, the Capital IQ stage fails gracefully. Yesterday's CIQ data stays in fundamentals.json and every other stage runs normally. You don't lose the whole refresh because one source had a bad day.
Scoring — how a ticker gets its label

Every ticker on the site ends up with a colored dot (green = underappreciated, blue = priced-in, amber = value-trap, red = avoid). Here's exactly what that dot is built from. All maths happen in jobs/scoring.py — no AI, no opaque models, just five fundamentals and five valuation checks, averaged.

First, the one bit of jargon: z-score

A z-score answers the question "how unusual is this number compared to the group?" Take everyone in NVDA's layer (GPU + accelerators) and look at their revenue-growth percentages. Compute the average and the typical spread. Then ask: how many "spreads" above (or below) the average is NVDA?

  • z = 0 → exactly average for the peer group
  • z = +1 → one "step" above average (top ~16% of peers)
  • z = +2 → strongly above (top ~2%)
  • z = −1 → one step below average

We z-score within the layer, not across the whole 66-ticker universe — that way memory companies are compared to memory companies, not to optics companies. Every z is clipped to ±3 so one freak number can't dominate. Convention: positive = good (strong growth, OR cheap valuation).

Layer roll-up — how a whole layer gets a status

Once every ticker has a quadrant, the layer takes the distribution and picks one label:

  • ≥ 30% of the layer's tickers are underappreciated → layer = Underappreciated (lowest bar — actionable case)
  • ≥ 50% are priced-in → Priced-in
  • ≥ 50% are value-trap → Value trap
  • ≥ 50% are avoid → Avoid
  • otherwise → Fair

If you want to pin a label manually (e.g. you know an area is being mispriced for a reason the data doesn't capture), set status_override in jobs/seed.py and that wins over the auto-status.

Bottleneck tightness score (0–100) · jobs/bottleneck.py

The little heat meter on each layer card. Higher = more evidence the layer is supply-constrained right now. Three ingredients, fixed weights:

  • 40% Signal density — how many signals (news, filings) tagged this layer in the last 90 days, ranked against other layers as a percentile.
  • 40% Tightness language — of those signals, how often do they contain phrases like "allocated", "sold out", "lead time", "shortage", "backlog", "supply constrained", "demand exceeds supply" (full list in the code). Hits-per-signal, percentile-ranked.
  • 20% Cross-quarter supply tightening — bonus +20 if any ticker in the layer had a recent earnings call where management said supply got tighter compared to the prior quarter.

The score is paired with up to 3 evidence quotes (the actual sentences that drove it up), shown in the tooltip and on the layer page.

Reverse-DCF · "what growth rate is the price implying?"

A regular DCF asks "given this growth assumption, what's fair value?" A reverse-DCF flips it: "given today's price, what growth rate would justify it?" Then you can compare that to what the company actually printed last year and ask: is the market expecting more, or less?

Two fixed assumptions, so every ticker is on the same playing field:

  • Discount rate r = 10% (what an equity investor demands)
  • Year-5 exit P/E = 18× (rough S&P long-run average)
  • 5-year horizon

The code solves for the EPS growth rate g that makes today's P/E exactly add up. If a stock is at 50× trailing earnings, the implied g might be ~30% per year. If the company just grew earnings at 12%, the gap (+18%) tells you the market is pricing in a re-acceleration. That gap is shown on stock.html and the stocks table as "implied vs recent growth".

Exit triggers · "when is the whole thesis wrong?"

The dashboard isn't here to make you bullish forever — it watches four pre-defined conditions that would break the inference-build thesis. Each lights up green (fine), amber (watch), or red (triggered).

  • Token-growth stalling — derived from OpenRouter daily snapshots. If frontier-inference token throughput stops growing, demand is rolling over.
  • Hyperscaler capex cut — driven by signals tagged with capex-guide-down language from MSFT / GOOGL / META / AMZN.
  • Architectural risk — count of arch-risk-tagged signals (Mamba, MoE, linear attention, KV-cache compression, distillation). High volume = the dominant transformer assumption is being challenged.
  • Taiwan / geopolitical — manually set by you in data/exit_triggers_manual.json. The only one not data-driven, because geopolitics doesn't show up in numbers until it's too late.

Operational notes
Atomicity

Every JSON write goes to file.json.tmp then is renamed into place. The browser never observes a half-written file. If a single stage fails, its prior data/*.json is preserved and the rest of the refresh continues.

Partial preservation

Curated signals (any id NOT prefixed rss- or edgar-) survive every refresh. --skip-ciq or a CIQ failure preserves prior ciq.* blocks. --skip-rss doesn't drop existing RSS rows; it just doesn't fetch new ones.

Daily cron

GitHub Actions (.github/workflows/refresh.yml) runs python -m jobs.refresh at 12:00 UTC on a hosted Ubuntu runner. SF_ACCOUNT / SF_USER / SF_PASS are repo secrets. The bot commits data/ as inference-dashboard-bot; Pages auto-redeploys on push.

Local run

Python is uv-managed. First time: uv sync. Then uv run python -m jobs.refresh and uv run python -m http.server 8000. Or double-click launch.command / refresh.command — both wrap uv run.
